# Слой 1 — Внесение данных собственником

## 1.1 Описание и цель

Слой 1 — точка входа данных в систему. Собственник взаимодействует только с Telegram-ботом: отправляет документы, надиктовывает или пишет текстом информацию о расходах и доходах. Система в ответ:

- Распознаёт содержимое через единый AI-провайдер (Google Gemini)
- **Автоматически определяет страну** (Беларусь или Польша) по признакам документа / текста
- Сохраняет файл на Google Drive в папку соответствующей страны
- Добавляет строку в реестр Google Sheets
- Подтверждает успешную обработку

Слой 1 работает **независимо от Слоя 2** — данные вносятся в течение всего месяца, сверка происходит отдельно.

---

## 1.2 Поддерживаемые форматы ввода

| Формат | Расширения / Тип | Язык | AI-движок |
|---|---|---|---|
| **Документ** | PDF, JPG, PNG | RU, BY, PL | Gemini Vision (`gemini-2.5-flash`) |
| **Голос** | Telegram voice message (.ogg) | RU, BY, PL | Gemini Audio → Gemini Text |
| **Текст** | Свободная форма в чате | RU, BY, PL | Gemini Text (`gemini-2.5-flash-lite`) |

> **MIME-нормализация.** Telegram иногда отдаёт файл с MIME-типом `application/octet-stream`. Система автоматически определяет реальный тип по имени файла и magic-bytes, приводит к одному из поддерживаемых (`application/pdf`, `image/jpeg`, `image/png`) и только после этого передаёт в Gemini Vision.

### Примеры корректного текстового ввода

```
# Беларусь
12 февраля 2026 оплачена аренда офиса 2500 EUR
Заплатил за рекламу в Google 5.02.2026 на сумму 430 USD
03.03.2026 закупка канцелярии для офиса 1200 BYN (не оплачено)
Поступила оплата от ООО Горизонт 15.01.2026 3000 BYN

# Польша (распознаётся по упоминанию польской фирмы / валюты PLN / языка)
Zapłaciłem za usługi sp. z o.o. Acme 14.02.2026 2500 PLN
Поступление от Nova Media sp. z o.o. 18.02.2026 4800 PLN
Оплата Allegro sp. z o.o. 03.03.2026 1200 PLN (не оплачено)
```

## 1.4 Логика обработки сообщения

```
ВХОДЯЩЕЕ СООБЩЕНИЕ
       │
       ├── ФАЙЛ (PDF / JPG / PNG)
       │       │
       │       ├─► MIME-нормализация (magic-bytes + имя файла)
       │       ├─► Gemini Vision: OCR + реквизиты + страна
       │       └── Нечитаем? → ❌ Сообщение об ошибке → СТОП
       │
       ├── ГОЛОС (.ogg)
       │       └─► Gemini Audio: STT → транскрипт
       │                               └─► Gemini Text: реквизиты + страна
       │
       └── ТЕКСТ (свободная форма)
               └─► Gemini Text: NLP-парсинг + страна

Все три пайплайна дают единый JSON-контракт → WF-04 (Drive) + WF-05 (Sheets)
```

**Ключевое:** страна определяется автоматически на этапе AI-парсинга, без дополнительных вопросов пользователю.

## 1.5 AI-парсинг реквизитов

### Модели Gemini

| Тип ввода | Модель | Обоснование |
|---|---|---|
| Документ (PDF / JPG / PNG) | `gemini-2.5-flash` (vision) | Мультимодальная, поддерживает OCR |
| Голос (.ogg) → текст | Gemini Audio | Единый провайдер, поддержка RU/BY/PL |
| Текст / транскрипт | `gemini-2.5-flash-lite` (text) | Дешевле vision-модели, достаточно для NLP |

### Единый JSON-контракт на выходе (для всех каналов)

```json
{
  "date_doc":       "DD.MM.YYYY или null",
  "counterparty":   "Название контрагента или null",
  "amount":         2500.0,
  "currency":       "BYN | USD | EUR | PLN | null",
  "date_paid":      "DD.MM.YYYY или null",
  "status":         "Поступление | Выбытие",
  "payment_status": "Оплачено | Не оплачено",
  "country":        "Беларусь | Польша",
  "description":    "Краткое резюме [пометки в скобках]"
}
```

**Сигнал ошибки:** `{"error": "НЕЧИТАЕМ"}` — запись не создаётся, файл не загружается.

### Определение Статуса (Поступление / Выбытие)

| Сигнал | Результат |
|---|---|
| Оплатил / заплатил / куплено / расход / оплата за / `zakupiono` / `zapłaciłem` | `Выбытие` |
| Поступила оплата / получено / доход / оплата от / `przychód` / `otrzymano` | `Поступление` |
| Тип документа = инвойс к оплате / расходная накладная / `faktura VAT` к оплате | `Выбытие` |
| Тип документа = приходный ордер / акт выполненных работ (входящий) | `Поступление` |

### Определение Статуса оплаты

| Сигнал | Результат |
|---|---|
| «оплачено» / `paid` / `zapłacono` / дата оплаты присутствует | `Оплачено` |
| «не оплачено» / «к оплате» / `unpaid` / `do zapłaty` | `Не оплачено` |
| Кассовый чек / фискальный / `paragon fiskalny` | `Оплачено` (всегда) |
| Неопределённость | `Не оплачено` (дефолт) |

### Определение страны (приоритетная таблица)

| Приоритет | Сигнал | Страна |
|---|---|---|
| 1 (сильный) | **NIP** (10 цифр, `XXX-XXX-XX-XX`), REGON, KRS | `Польша` |
| 1 (сильный) | **УНП** (9 цифр) | `Беларусь` |
| 2 | Юр. форма: `sp. z o.o.`, `S.A.`, `sp.j.`, `sp.k.` | `Польша` |
| 2 | Юр. форма: `ОАО`, `ЗАО`, `ОДО`, `ЧУП`, `УП`, `ИП` | `Беларусь` |
| 3 | Польский язык: `faktura`, `nabywca`, `netto`, `brutto`, `do zapłaty` | `Польша` |
| 3 | Русский/белорусский: `накладная`, `счёт-фактура`, `акт вып. работ` | `Беларусь` |
| 4 | Польский индекс `XX-XXX`, польские города | `Польша` |
| 4 | Белорусские адреса (Минск, Гомель, Брест…) | `Беларусь` |
| 5 (слабый) | Валюта PLN | `Польша` (подсказка) |
| 5 (слабый) | Валюта BYN | `Беларусь` (подсказка) |
| — | Нет однозначных признаков | `Беларусь` (дефолт) |

**USD и EUR страну не определяют** — обе юрисдикции используют эти валюты.

**Конфликт признаков BY↔PL:** страна = по владельцу/продавцу документа. Пометка `[смешанная сделка BY↔PL]` в description.

**Слабый сигнал:** добавить `[страна определена эвристически]` в description.

### Формирование поля «Описание AI»

```
Структура: {тип}. {назначение}[, {контрагент}]. {сумма} {валюта}[. {пометки}]

Пример 1 (BY): «Выбытие. Оплата аренды офиса, февраль 2026. 2500 EUR»
Пример 2 (BY): «Выбытие. Реклама Google Ads. 430 USD»
Пример 3 (BY): «Выбытие. Закупка канцелярии. 1200 BYN. [Не оплачено]»
Пример 4 (BY): «Поступление. Оплата от ООО Горизонт. 3000 BYN»
Пример 5 (PL): «Выбытие. Usługi marketingowe, Acme sp. z o.o. 2500 PLN»
Пример 6 (PL): «Поступление. Faktura VAT od Nova Media sp. z o.o. 4800 PLN»
Пример 7 (mixed): «Выбытие. Оплата PL-контрагенту. 500 EUR. [смешанная сделка BY↔PL]»
```

## 1.6 Схема данных — Реестр

### Финальная структура (11 видимых + 1 скрытая = 12 колонок) — **обновлено в v2.3**

| # | Колонка | Тип | Правило | Пример |
|---|---|---|---|---|
| A | **Nr ID** | String | `{тип}.{NNNN}`. `0`=Поступление, `1`=Выбытие. Счётчики независимы по `(Страна, Тип)`, сбрасываются 1-го числа месяца | `1.0003` |
| B | **Статус** | Enum | `Поступление` \| `Выбытие` | `Выбытие` |
| C | **Дата документа** | DD.MM.YYYY | Из AI. Если не определена — пусто | `05.02.2026` |
| D | **Контрагент** | String | Из AI. Если не определён — пусто | `Acme sp. z o.o.` |
| E | **Дата оплаты** | DD.MM.YYYY | Из AI. Если `Статус оплаты = Не оплачено` — пусто | `05.02.2026` |
| F | **Сумма** | Number | Число со знаком: `+` для Поступления, `−` для Выбытия | `-2500` |
| G | **Валюта** | Enum | `BYN` \| `USD` \| `EUR` \| `PLN` | `PLN` |
| H | **Статус оплаты** | Enum | `Оплачено` \| `Не оплачено` | `Оплачено` |
| I | **Страна** | Enum | `Беларусь` \| `Польша`. Никогда не пусто (NFR-14) | `Польша` |
| J | **Ссылка на документ** | URL | Google Drive URL. При текст/голос — пусто | `https://drive.google.com/…` |
| K | **Описание AI** | Text | Краткое резюме + пометки в `[скобках]` | `Выбытие. Usługi marketingowe. 2500 PLN` |
| L *(скрытый)* | **UUID** | UUID v4 | Глобально уникальный технический ID. Используется Слоем 2 как ключ матчинга | `550e8400-e29b-…` |

> **Почему разделили Сумму и Валюту (изменение v2.3):** Слой 2 сравнивает суммы числовым сравнением `===`, а не парсингом строки. Это исключает класс ошибок при сверке.

### Правила Nr ID

```
Тип 0 = Поступление, Тип 1 = Выбытие

Счётчики независимы по (СТРАНА, ТИП) и сбрасываются 1-го числа каждого месяца.
В одном месяце могут существовать одновременно BY:0.0001 и PL:0.0001 — это разные записи.

Пример (Февраль 2026):
  Беларусь                    Польша
  ────────                    ──────
  0.0001  ← Поступление BY    0.0001  ← Поступление PL
  1.0001  ← Выбытие BY        1.0001  ← Выбытие PL
  1.0002  ← Выбытие BY        1.0002  ← Выбытие PL
```

**Уникальность:** Nr ID уникален внутри тройки `(Страна, YYYY.MM, Тип)`. Для глобальной уникальности — скрытый **UUID** (колонка L).

### UUID — роль в системе

UUID генерируется однократно при создании записи (WF-05) и никогда не меняется. Не отображается пользователю. Когда Слой 2 находит соответствие между транзакцией банка и строкой реестра, UUID этой строки записывается в лист «Банк» (колонка M) как подтверждение связи. Таким образом, Слой 2 **не модифицирует основной реестр** — UUID служит только ссылкой в обратном направлении.

## 1.7 Хранение на Google Drive

### Иерархия папок

```
Google Drive (DRIVE_ROOT_FOLDER_ID)/
├── Belarus/
│   ├── 2026.01/
│   │   ├── income/
│   │   │   └── 2026.01_ooo_gorizont_3000.pdf
│   │   └── outcome/
│   │       └── 2026.01_google_llc_430.jpg
│   └── 2026.02/
│       ├── income/
│       └── outcome/
└── Poland/
    ├── 2026.01/
    │   ├── income/
    │   │   └── 2026.01_nova_media_sp_z_o_o_4800.pdf
    │   └── outcome/
    └── 2026.02/
        ├── income/
        └── outcome/
```

Выбор корневой папки (`Belarus/` vs `Poland/`) определяется значением поля **Страна**. Подпапка `income` / `outcome` — из поля **Статус**.

### Правила формирования имени файла

```
Формат: {YYYY.MM}_{контрагент_транслит}_{abs(Сумма)}.{расширение}

Правила нормализации:
  Кириллица   → транслит ГОСТ 7.79
  Диакритика  → убрать (ł→l, ś→s, ą→a, ę→e, ó→o, ź→z, ż→z, ć→c, ń→n)
  Пробелы     → _
  Точки (sp.) → удалить
  Спецсимволы → удалить
  Регистр     → нижний
  Нет контрагента → «unknown»
  Сумма в имени → abs(Сумма), без знака

Примеры:
  2026.01_ooo_gorizont_3000.pdf          ← BY
  2026.02_google_llc_430.jpg             ← BY
  2026.03_unknown_1200.pdf               ← BY без контрагента
  2026.01_nova_media_sp_z_o_o_4800.pdf   ← PL
  2026.02_acme_sp_z_o_o_2500.pdf         ← PL
```

## 1.8 Запись в Google Sheets

### Последовательность операций (WF-05)

```
1. Определить период (YYYY.MM) из date_doc или текущей даты
2. Получить счётчик из листа «Счётчики» по ключу (Период, Страна)
3. Инкрементировать нужный счётчик (Поступления или Выбытия)
4. Сформировать Nr ID
5. Если файл → загрузить на Drive → получить URL
6. Сформировать строку: 11 полей (A–K) + UUID (L)
7. Append строки в лист «Реестр»
8. Обновить счётчик в листе «Счётчики»
9. Отправить подтверждение в Telegram
```

### Шаблон подтверждения пользователю

```
✅ Записано

📌 1.0003 | Выбытие
🇵🇱 Польша
📅 05.02.2026
🏢 Acme sp. z o.o.
💰 -2500 PLN | Оплачено
📁 Poland/2026.02/outcome/
🔗 [Ссылка на файл]

💬 Выбытие. Usługi marketingowe, Acme sp. z o.o. 2500 PLN
```

> **Флаг страны** — важный UX-сигнал: пользователь сразу видит, правильно ли AI определил страну.

### MIME-нормализация (NFR-11)

```javascript
// Magic bytes для определения реального типа
PDF:  [0x25, 0x50, 0x44, 0x46]  → application/pdf
JPEG: [0xFF, 0xD8, 0xFF]        → image/jpeg
PNG:  [0x89, 0x50, 0x4E, 0x47]  → image/png

// Если тип не определён → {"error": "НЕЧИТАЕМ"}
```

### Парсинг JSON-ответа Gemini (NFR-12)

```javascript
function parseGeminiResponse(text) {
  const cleaned = text.replace(/```json\s*/gi, '').replace(/```\s*/g, '').trim();
  const match = cleaned.match(/\{[\s\S]*\}/);
  if (!match) throw new Error('JSON not found in Gemini response');
  return JSON.parse(match[0]);
}
```

# Слой 2 — Сверка и уведомления

## 2.1 Описание и цель

Слой 2 сопоставляет транзакции из банковских выписок с документами реестра. В MVP — **полная двусторонняя сверка по Беларуси и Польше одновременно**.

Источник банковских данных — реальные выписки, загружаемые командой `/vpsk`. CSV-заглушка упразднена.

**Принцип изоляции:** Слой 2 **читает** основной реестр и **не изменяет** его. Все результаты хранятся на отдельных листах: **«Банк»** и **«Сверка»**.

**UUID как связующее звено:** при матчинге UUID записи реестра записывается в лист «Банк» (колонка M). Так образуется ссылка банк → реестр без модификации реестра.

**Gemini для парсинга выписок:** банковские выписки (Приорбанк DOCX, PKO BP PDF) содержат контрагентов в многострочном свободном тексте — это невозможно надёжно разобрать детерминированным кодом. Gemini парсит выписку один раз и нормализует данные в единый формат, идентичный полям реестра. После этого матчинг — чистая математика.

## 2.4 Лист «Банк» — структура и наполнение

Лист содержит нормализованные транзакции из всех загруженных выписок. Заполняется WF-06. **Ключевые поля D–J идентичны полям реестра по названию и формату** — основа детерминированного матчинга.

| # | Колонка | Тип | Примечание |
|---|---|---|---|
| A | **bank_id** | String | Авто: `{BY\|PL}_{YYYYMM}_{seq:04}`, напр. `PL_202602_0003` |
| B | **Банк** | Enum | `Priorbank` \| `PKO_BP` |
| C | **Период** | String | `YYYY.MM` |
| D | **Страна** | Enum | `Беларусь` \| `Польша` ← совпадает с реестром |
| E | **Статус** | Enum | `Поступление` \| `Выбытие` ← совпадает с реестром |
| F | **Дата документа** | DD.MM.YYYY | ← совпадает с реестром |
| G | **Контрагент** | String | ← совпадает с реестром |
| H | **Дата оплаты** | DD.MM.YYYY | ← совпадает с реестром |
| I | **Сумма** | Number | Всегда положительная ← совпадает с abs(реестр.Сумма) |
| J | **Валюта** | Enum | `BYN\|USD\|EUR\|PLN` ← совпадает с реестром |
| K | **Описание** | Text | Исходный текст из выписки |
| L | **Статус сверки** | Enum | `PENDING` \| `MATCHED` \| `UNMATCHED` |
| M | **UUID реестра** | String | UUID строки реестра при MATCHED, иначе пусто |

> **Примечание по знаку суммы:** в реестре `Сумма` содержит знак (±). В листе «Банк» `Сумма` всегда положительная — знак определяется через поле `Статус`. При матчинге: `abs(реестр.Сумма) === банк.Сумма`.

## 2.5 Алгоритм матчинга

Запускается WF-07 по команде `/reconcile` или автоматически (cron 1-е число, 09:00 UTC+3).

### Входные данные

```
• Лист «Банк»: строки за отчётный период (Статус сверки = PENDING или UNMATCHED)

• Лист «Реестр»: строки за отчётный период
  Фильтр: Статус оплаты = «Оплачено» ИЛИ Дата оплаты не пустая.
  Записи «Не оплачено» исключаются — нет соответствующей банковской проводки.
```

### Алгоритм (для каждой транзакции из листа «Банк»)

```
Шаг 0. Жёсткий фильтр по стране и типу (обязательный)
        Банк.Страна  == Реестр.Страна
        Банк.Статус  == Реестр.Статус
        → 0 кандидатов → UNMATCHED, стоп

Шаг 1. Жёсткий фильтр по сумме и валюте (обязательный)
        abs(Реестр.Сумма)  === Банк.Сумма   (числовое ===)
        Реестр.Валюта      === Банк.Валюта   (строковое, регистронезависимо)
        → 0 совпадений → UNMATCHED, стоп

Шаг 2. Фильтр по дате (±3 дня, мягкий)
        Базовая дата реестра:
          если Реестр.«Дата оплаты» не пусто → использовать её
          иначе если Реестр.«Дата документа» не пусто → использовать её
          иначе → шаг пропускается (не блокирует)
        |Банк.«Дата оплаты» − Реестр.базовая_дата| ≤ 3 дня

Шаг 3. Нечёткий контрагент (мягкий, подтверждение)
        Если оба поля не пустые:
          similarity(Банк.Контрагент, Реестр.Контрагент) ≥ 70%
          (trigram или Levenshtein)
        Несовпадение не блокирует, только снижает приоритет.

Результат:
  Ровно 1 кандидат     → MATCHED; UUID реестра → Банк.M
  Несколько кандидатов → MATCHED по max схожести;
                         + пометка [ambiguous] в Банк.Описание
  0 кандидатов         → UNMATCHED
```

### Граничные случаи

| Ситуация | Поведение |
|---|---|
| Транзакция без контрагента (банк) | Шаг 3 пропускается |
| Реестр: нет ни одной даты | Шаг 2 пропускается |
| Реестр: `Статус оплаты = Не оплачено` | Запись не участвует в матчинге |
| Запись из текста/голоса (нет файла) | Участвует в матчинге по имеющимся полям |
| Несколько документов на одну сумму | Берётся наиболее близкий по дате |

### Пример кода матчинга (WF-07, JS-нода)

```javascript
// bankRows — строки листа «Банк» (PENDING/UNMATCHED за период)
// registryRows — строки листа «Реестр» (Оплачено за период)

for (const tx of bankRows) {
  // Шаги 0 + 1: жёсткий фильтр
  let candidates = registryRows.filter(r =>
    r['Страна']  === tx['Страна']  &&
    r['Статус']  === tx['Статус']  &&
    Math.abs(parseFloat(r['Сумма'])) === parseFloat(tx['Сумма']) &&
    r['Валюта'].toUpperCase() === tx['Валюта'].toUpperCase()
  );

  // Шаг 2: дата ±3 дня
  if (candidates.length > 1) {
    candidates = candidates.filter(r => {
      const base = r['Дата оплаты'] || r['Дата документа'];
      if (!base) return true;
      return Math.abs(daysDiff(tx['Дата оплаты'], base)) <= 3;
    });
  }

  // Шаг 3: нечёткий контрагент
  if (candidates.length > 1 && tx['Контрагент']) {
    candidates = candidates
      .filter(r => !r['Контрагент'] ||
        similarity(tx['Контрагент'], r['Контрагент']) >= 0.70)
      .sort((a, b) =>
        similarity(tx['Контрагент'], b['Контрагент']) -
        similarity(tx['Контрагент'], a['Контрагент'])
      );
  }

  if (candidates.length >= 1) {
    tx['Статус сверки'] = 'MATCHED';
    tx['UUID реестра']  = candidates[0]['UUID'];
    if (candidates.length > 1) tx['Описание'] += ' [ambiguous]';
  } else {
    tx['Статус сверки'] = 'UNMATCHED';
  }
}
```

## 2.9 User Stories — Слой 2

**US-06 — Загрузка выписки**
> Как собственник, я отправляю банковскую выписку командой /vpsk и получаю подтверждение о количестве загруженных транзакций.

**Критерии приёмки:**
- Поддерживаются: Приорбанк `.docx`, PKO Bank `.pdf`
- Gemini корректно парсит транзакции в единый формат
- Данные записаны в лист «Банк» со статусом PENDING
- Повторная загрузка того же периода заменяет старые данные без дублей
- Подтверждение: страна, банк, период, количество транзакций, суммарные поступления/выбытия

**US-07 — Автоматическая сверка**
> Как система, 1-го числа каждого месяца я автоматически сверяю транзакции банков с документами реестра.

**Критерии приёмки:**
- Cron запускается в 09:00 UTC+3 ± 1 минута
- Сверка выполнена для BY и PL
- Лист «Сверка» перезаписан
- Уведомление отправлено всем получателям

**US-08 — Ручная сверка**
> Как собственник, я запускаю сверку командой /reconcile в любое время.

**Критерии приёмки:**
- `/reconcile` — текущий месяц, BY + PL
- `/reconcile 2026-02` — конкретный месяц
- Результат аналогичен автосверке
- Если сверка уже запущена — бот сообщает об этом

**US-09 — Уведомление о результатах**
> Как собственник/бухгалтер, я получаю структурированный итог с разбивкой по странам.

**Критерии приёмки:**
- Уведомление содержит секции BY и PL раздельно
- Перечислены незакрытые позиции: страна, дата, сумма, контрагент
- При полном закрытии — «Месяц закрыт без замечаний»
- Ссылка на Google Sheets

**US-10 — Просмотр незакрытых позиций**
> Как собственник, я вижу актуальный список командой /missing без открытия Sheets.

**Критерии приёмки:**
- Список из последней сверки, BY + PL раздельно
- Если сверки не было — бот сообщает об этом
- Каждая позиция: флаг страны, дата, сумма, контрагент

# AI-подсистема — общая схема

### Распределение AI-нагрузки (обновлено в v2.3)

```
ВХОД               ПРЕПРОЦЕССИНГ      AI-ДВИЖОК         ВЫХОД
────               ─────────────      ──────────        ─────

Слой 1:
PDF/JPG/PNG ──► MIME-нормализация ─┐
                                    ├──► GEMINI ───► JSON-контракт
Voice (.ogg) ─► Gemini Audio ──────┤    Vision /      (9 полей)
                                    │    Text
Text ─────────► (без препроц.) ────┘

Слой 2:
DOCX/PDF ────► MIME-нормализация ──► GEMINI ───► JSON[] транзакций
(выписка)                           Vision       (нормализованные)
```

### Принципы

1. **Единый провайдер.** Google Gemini для всех каналов. Anthropic Claude выведен из MVP начиная с v2.2.
2. **Единый контракт (Слой 1).** Один JSON-объект независимо от канала. Downstream-логика провайдеро-независима.
3. **Единый стандарт полей (Слой 2).** Парсинг выписки возвращает поля с теми же именами и форматами, что и реестр. Матчинг — прямое сравнение без трансформаций.
4. **Единый сигнал ошибки.** `{"error": "НЕЧИТАЕМ"}` для документов Слоя 1. `{"error": "NOT_STATEMENT"}` для выписок Слоя 2.
5. **Единая точка отказа.** Сбой Gemini блокирует все каналы Слоя 1 и парсинг выписок Слоя 2. Алерты на 4xx/5xx обязательны.

### Операционные требования

- Один набор ключей: **`GEMINI_API_KEY`**
- Мониторинг: алерты на ошибки Gemini; публикация статуса в боте при недоступности
- При недоступности Gemini: ручной fallback — смена credential / модели через обновление переменных n8n

# Глоссарий

| Термин | Определение |
|---|---|
| **Слой 1** | Модуль ежедневного внесения документов (WF-01..WF-05) |
| **Слой 2** | Модуль загрузки выписок и ежемесячной сверки (WF-06..WF-07) |
| **Nr ID** | Идентификатор записи вида `0.0001` / `1.0001`. Уникален внутри `(Страна, YYYY.MM, Тип)`. Сбрасывается ежемесячно |
| **UUID** | Глобально уникальный технический ID записи реестра (столбец L, скрытый). Используется Слоем 2 как ключ ссылки из листа «Банк» на строку реестра без модификации реестра |
| **bank_id** | Технический ID строки в листе «Банк». Формат: `{BY\|PL}_{YYYYMM}_{seq:04}` |
| **Матчинг** | Детерминированный алгоритм сопоставления транзакции из листа «Банк» с записью реестра. 4 шага: страна+тип → сумма+валюта → дата±3дня → контрагент fuzzy |
| **Лист «Реестр»** | Основной лист Google Sheets. Хранит все документы Слоя 1. Слой 2 только читает его |
| **Лист «Счётчики»** | Сервисный лист. Хранит счётчики Nr ID с ключом `(Период, Страна)` |
| **Лист «Банк»** | Лист с нормализованными транзакциями из загруженных выписок. Заполняется WF-06 |
| **Лист «Сверка»** | Лист с результатами матчинга (MATCHED/UNMATCHED). Перезаписывается WF-07 |
| **PENDING** | Статус строки «Банк»: транзакция ещё не участвовала в сверке |
| **MATCHED** | Транзакция сопоставлена с записью реестра; UUID реестра записан в столбец M |
| **UNMATCHED** | Транзакция не нашла соответствия в реестре |
